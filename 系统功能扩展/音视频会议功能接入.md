# 音视频会议功能接入

---

### 架构选型

在现有IM架构上音视频会议功能，最稳线路是 `webrtc` + 独立rtc服务域，不将流媒体服务塞进现有的msg_server中，

架构选型：

1. 信令层：沿用现有im通道（msg_server、route_server）承载呼叫状态、房间控制等
2. 媒体层：独立rtc集群（sfu为主）
3. NAT穿透：标准 stun/turn
4. 会议规模：
   - 1v1小群（<=8）：可以先从简单SFU起步
   - 中大型会议：必须SFU，后续增加级联/分区

推荐优先级：

1. webrtc + 自建 sfu（mediasoup/janus/pion）
2. 快速上线可选用云 RTC，腾讯云 TRTC/ 声网 Agora先跑业务闭环

不建议自研媒体协议：

1. 音视频核心难点在于网络抖动、丢包、回声、拥塞控制、设备兼容
2. webrtc生态已经把这些坑踩完，生产风险最低

### 技术落地

结合现有teamtalk的落地分层，

- im（已有）：好友、群、文本消息、离线推送
- rtc-signaling（新增）：发起\应答\挂断、房间成员、设备状态
- rtc-media（新增）：sfu媒体转发
- rtc-control（新增）：鉴权、房间策略、录制\旁路推流编排
- push（现有扩展）：来电通知（国内安卓厂商+APNS）

#### 关键技术选型点

- 拓扑
  - P2P：仅 1v1 可考虑，不适合会议
  - SFU：会议主流（推荐）
  - MCU：成本高、延迟高，非必要不选
- 客户端 SDK
  - 移动端：原生 WebRTC SDK（iOS/Android）
  - 桌面端：WebRTC Native / Electron（按你客户端技术栈）
- 编解码
  - 视频：H264 起步（兼容好），后续可加 VP9/AV1
  - 音频：Opus
- 弱网策略
  - Simulcast / SVC
  - 自适应码率、降分辨率
- 安全
  - DTLS-SRTP（媒体加密）
  - TURN 鉴权（短期 token）
  - 房间入会 token（服务端签发）

#### 开发步骤

现有架构下进行开发步骤：

1. 定义RTC信令协议（关键）
   - `call_invite / call_accept / call_reject / call_end` 
   - `room_join / room_leave / participant_update` 
   - `ice_candidate / sdp_offer / sdp_answer` 
2. 单独起个 `rtc_server` 
   - 只做信令网关 + SFU对接，不污染现有的消息服务 `msg_server` 
3. 先做1v1音频闭环：
   - 打通鉴权、呼叫、建链、挂断、重连
4. 再上多人会议
   - 小规模会议、说话人检测、屏幕共享、录制等





### 相关技术自学

主要是调用webrtc api，对系统架构、网络通信、媒体处理和用户体验的综合工程。学习路径：

- 源码地址：https://webrtc.googlesource.com/src/+/refs/heads/main
- webrtcsdk：https://github.com/webrtc-sdk/webrtc

#### 第一阶段：地基与理论扫盲 (1-2周)

开始前，先建立起对WebRTC整体架构的清晰认知，帮助后续能看懂那些复杂的模块是如何一起工作的。

- **核心认知**：通过一份清晰的"三层模型"架构图，快速建立对WebRTC的整体认知。理解核心API（`RTCPeerConnection`, `MediaStream`, `RTCDataChannel`）。
- **关键组件**：理解 WebRTC 架构中的四大基本服务器，这是实现全网前后端联调的基础。
  - **信令服务器 (Signaling Server)**：这是你需要用 Node.js, Go 等服务端语言来开发的部分，负责转发房间号、会话描述协议（SDP）等信令。
  - **STUN 服务器**：提供获取设备公网 IP 和端口的能力，是 NAT 网络穿透的第一道防线。
  - **TURN 服务器**：当 P2P 连接失败时的“终极武器”，负责中继所有音视频数据。建议从社区版开源项目（如 `coturn`）开始部署。
  - **媒体服务器 (Media Server)**：当你需要支持 3 人以上的音视频通话时就必须引入的组件。负责选择性转发（SFU，最主流）或多路混流。

#### 第二阶段：从零手写1对1通话 (2-3周)

动手搭建最小化可行产品（MVP），把知识“跑”起来，加深理解。

- **小目标A：打造你的信令服务器**
  - **技术栈**：选你最熟悉的语言（如 Node.js + Express/[Socket.IO](https://socket.io/)，Go, Python）创建一个 WebSocket 服务器。
  - **核心功能**：实现简单的“房间”逻辑，负责转发客户端的 `offer`, `answer`, `candidate`。
- **小目标B：连接客户端**
  - **应用**：用 HTML/JS 创建一个简单的 Web 应用，分别用在两个浏览器标签或两台电脑上。
  - **流程**：实现用户点击按钮后的完整通话流程：`getUserMedia` -> 创建 `PeerConnection` -> 生成 `offer` -> 通过你的信令服务器发送给另一端。
- **小目标C：测试与穿透**
  - **环境**：突破本地局域网，在真实网络环境下进行测试。
  - **配置**：在你的代码中添加 Google 公共 STUN 服务器 (`stun:stun.l.google.com:19302`) 进行测试，并尝试搭建并配置开源 TURN 服务器，理解 P2P 失败时媒体流如何自动回退到 TURN 中继。

#### 第三阶段：进阶路线选择

当 1v1 通话稳定运行后，你可以根据产品需求选择下一阶段：

- **A线: 构建你自己的小型SFU**：适合希望完全掌控技术栈、团队实力强劲的场景。使用 Pion (Go)、Mediasoup (Node.js) 等框架，从零搭建一个支持多人的 SFU，实现 3方及以上的人数通话，这将极大锻炼你的后端架构能力。
- **B线: 评估与集成商用的RTC SDK**：适合希望快速上线功能、保持轻资产的场景。调研声网Agora、腾讯云实时音视频（TRTC）等云厂商的方案。将 WebRTC 作为底层，将集成工作重点放在 API 调用和业务逻辑联调上，极大缩短开发周期。

#### 第四阶段：深度修炼 (长期)

- **源码研读**：深入 WebRTC 官方源码，重点关注 `pc/`, `call/`, `media/`, `modules/`。
- **跟进前沿**：关注 WebCodecs, WebTransport 等新 API，探索 WebRTC 的新应用场景。

| 阶段        | 核心目标     | 关键产出                                                     |
| :---------- | :----------- | :----------------------------------------------------------- |
| **第1-2周** | 完成理论基础 | 能清晰讲述WebRTC的架构、核心API和四大服务器组件。            |
| **第3-4周** | 跑通1对1通话 | 一个基于信令服务器的可用于局域网内1对1通话的Demo。           |
| **第5-6周** | 应对外网难题 | 部署并可配置STUN/TURN服务器，使Demo能在不同网络环境下建立连接。 |
| **第7周起** | 选择进阶方向 | 选择搭建一个简单的SFU或集成一个商用的RTC SDK。               |
